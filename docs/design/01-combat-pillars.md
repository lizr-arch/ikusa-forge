# Combat Pillars v0.1

## 1. Pre-battle construction matters more than micro-control

The player wins mostly through:

- unit selection
- weapon selection
- formation
- synergy
- matchup reading

Battle should mostly run automatically.

## 2. Automatic battle must be explainable

The player should understand:

- who dealt damage
- who absorbed damage
- which skill triggered
- which formation bonus mattered
- which enemy threat caused the loss

A battle that only says “Victory” or “Defeat” is not enough.

## 3. Formation must change results

Changing grid positions should affect:

- first contact
- target selection
- protection of backline units
- area skill impact
- adjacency bonuses
- unit survival

## 4. Build pieces must combine

Units, weapons, skills, tags, and formations should not be isolated.

Examples:

- spear units gain brace counter in front row
- archers gain bonus if protected by shield units
- dual katana units gain burst if placed on flank
- banner support buffs adjacent ashigaru

## 5. MVP should be small but expressive

The first version should use:

- 4x3 battlefield
- 6 player units
- 6 enemy units
- 5 weapon types
- 10 skills
- 3 formations
- 6 synergies

Depth comes from interaction, not content volume.

## 6. Debuggability is a game feature

The development tool should expose:

- replay timeline
- damage report
- skill trigger report
- target choice reasons
- key moments
- config hash and seed
